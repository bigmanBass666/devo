# 心跳协议 — Heartbeat Protocol

> **版本**: v0.5.0
> **触发条件**: 用户输入 `/heartbeat` 或在新会话中粘贴心跳指令模板
> **依赖**: 必须先加载 AGENTS.md（基础规则）+ valveos-protocol.md（ValveOS 协议）
>
> ⚠️ **本文件定义 ValveOS 心跳模式下的通信与行为规范。**
> 心跳模式是 ValveOS 的增强模式，允许 Agent 自主轮询 inbox 并直接通信。
> 非心跳模式下，铁门协议仍适用（详见 valveos-protocol.md）。

---

## 概述

心跳协议解决 ValveOS 的核心瓶颈——"通信阀门"。通过让 Agent 执行轮询循环（Sleep → Read → Process → Respond → Loop），Agent 可以自主检测并响应 inbox 消息，无需用户中转。

**核心突破**：AI 可以执行 `Start-Sleep -Seconds N`（Windows）或 `sleep N`（Unix）后继续操作，实现非阻塞式等待 + 循环检查。

**类比**：从"对讲机模式"（必须按住按钮才能说话）升级为"电话模式"（双方始终在线，随时对话）。

## 🔄 心跳轮询循环

### 标准循环

```
┌─────────────────────────────────────────────┐
│              心跳轮询循环                     │
│                                             │
│  1. 等待新消息：执行 Sleep(N秒)              │
│  2.【view_files】读取自己的 inbox 文件        │
│  3. 检测是否有未处理消息（无 ✅ 标记的）      │
│  4. 如有 → 处理消息 → 标记 ✅ → 回复        │
│  5. 更新 heartbeat-panel.md（计数+1）        │
│  6. 本轮完成 → 重复执行步骤 1                │
│                                             │
│  ⚠️ 不使用 while/for 循环                   │
│  ⚠️ 不挂到后台运行                          │
│  ⚠️ 在前景等待                              │
│  ⚠️ 每轮从 Sleep 开始                       │
│  ⚠️ 输出只含处理结果，不含机制评论            │
│  ⚠️ view_files 穿插在 Sleep 之间            │
└─────────────────────────────────────────────┘
```

### 轮询间隔

| Agent 类型 | 默认间隔 | 原因 |
|-----------|---------|------|
| 🔴 常驻心跳 | 240 秒 | 需要快速响应 |
| 🟡 按需心跳 | 400 秒 | 有任务时启动，可稍慢 |
| 🔄 周期心跳 | 800 秒 | 定期巡检，不需要实时 |
| 🎯 监督心跳 | 640 秒 | 监控异常，中等频率 |

### 关键约束

1. **禁止循环语句**：不使用 while/for，每次循环由 AI 自主重复执行
2. **禁止后台运行**：必须在前景等待，确保 AI 能感知每次循环
3. **单一 Sleep 命令**：只执行给定的 Sleep 命令，不自行编造其他等待方式
4. **平台适配**：Windows 用 `Start-Sleep -Seconds N`，Unix 用 `sleep N`
5. **必须使用 `view_files` 工具读取 inbox 文件**：禁止使用 `run_command`（如 `cat`/`type`）读取文件，因为 Trae 会将重复的 `run_command` 判定为重复调用并禁止执行。`view_files` 不受此限制，可无限次重复调用
6. **每轮从 Sleep 开始**：Sleep 是"等待新消息"的工具，不是无意义的等待
7. **输出只含处理结果**：只输出消息处理、任务分配等结果，不输出关于轮询机制本身的评论
8. **view_files 穿插在 Sleep 之间**：防止 Trae 拦截连续的 run_command

## 🔄 Agent 生命周期状态机

```
Dormant ──(用户启动心跳)──→ Heartbeat ──(检测到任务)──→ Working ──(任务完成)──→ Standby
   ↑                           │                          │                       │
   │                           │←──(无任务，继续轮询)──────┘                       │
   │                           │←──────────(新任务到来)──────────────────────────┘
   │                           │
   └────(用户停止心跳)─────────┘
```

### 状态定义

| 状态 | 含义 | 行为 |
|------|------|------|
| 🌙 Dormant | 未启动心跳 | 不轮询，需要用户唤醒 |
| 💓 Heartbeat | 轮询中，等待任务 | 执行轮询循环，检测新消息 |
| ⚡ Working | 正在处理任务 | 执行具体工作，处理完回到 Standby |
| 💤 Standby | 空闲轮询 | 继续轮询，等待新任务 |

### 状态转换规则

| 从 → 到 | 触发条件 | 动作 |
|---------|---------|------|
| Dormant → Heartbeat | 用户粘贴心跳指令 | 初始化，开始轮询 |
| Heartbeat → Working | 检测到未处理消息 | 处理消息，执行任务 |
| Working → Standby | 任务完成 | 标记消息 ✅，继续轮询 |
| Standby → Working | 检测到新消息 | 处理新消息 |
| Standby → Heartbeat | 无（等价状态） | 继续轮询 |
| 任意 → Dormant | 用户停止心跳或会话结束 | 更新面板状态 |

## 📨 结构化消息格式

### 消息结构

每条 inbox 消息必须包含以下字段：

```markdown
## 📨 MSG-[ID] | From: [发送者] | Type: [类型] | [时间戳]

[消息内容]

---
```

### 字段说明

| 字段 | 格式 | 示例 |
|------|------|------|
| ID | MSG-NNN（自增） | MSG-001 |
| 发送者 | Agent 名称 | Coordinator |
| 类型 | task/response/status/query/shutdown | task |
| 时间戳 | yyyy-MM-dd HH:mm:ss | 2026-04-23 14:30:00 |

### 消息类型

| 类型 | 用途 | 示例 |
|------|------|------|
| task | 任务分配 | Coordinator 分配子任务给 Worker |
| response | 任务响应 | Worker 完成后通知 Coordinator |
| status | 状态更新 | Agent 报告当前进度 |
| query | 信息查询 | Agent 向其他 Agent 询问信息 |
| shutdown | 停止信号 | 用户或 COO 通知 Agent 停止心跳 |

### 消息处理标记

- 未处理：消息头部无 ✅
- 已处理：消息头部添加 ✅

```markdown
## 📨 MSG-001 | From: Coordinator | Type: task | 2026-04-23 14:30:00 ✅

[已处理的消息内容]

---
```

### 消息写入规则

1. 追加到 inbox 文件末尾（不覆盖已有消息）
2. 每条消息用 `---` 分隔
3. 消息 ID 在同一 inbox 内自增
4. 时间戳使用 `$NOW` 变量（与 valveos-protocol.md 时间纪律一致）

## 🛡️ 护栏原则

> 替代阀门原则第 1-2 条（仅心跳模式下）。唤醒模式下阀门原则仍适用。

### 核心条文

1. **通信自由**：Agent 可以直接写入其他 Agent 的 inbox，无需用户中转
2. **安全边界不可逾越**：社交边界、安全铁律始终生效（见 AGENTS.md）
3. **用户最高权限**：用户可随时介入任何 Agent 的会话，中断轮询
4. **停止信号必须响应**：心跳模式下的 Agent 收到 shutdown 类型消息必须停止轮询

### 与安全铁律的关系

| 安全铁律 | 心跳模式下是否适用 | 说明 |
|---------|------------------|------|
| 永不删除 main 分支 | ✅ 适用 | 无论何种模式 |
| 永不 push 到 upstream | ✅ 适用 | 无论何种模式 |
| 永不向 PR 包含 tasks/ 等 | ✅ 适用 | 无论何种模式 |
| 社交边界 | ✅ 适用 | 心跳不改变权限边界 |

### 用户介入方式

1. **直接在会话中输入**：Agent 立即响应用户指令
2. **发送 shutdown 消息**：在 Agent 的 inbox 中写入 shutdown 类型消息
3. **关闭会话**：Agent 自然停止

## 🚨 错误恢复机制

### Sleep 命令失败

```
1. 记录错误到日志
2. 等待 2 秒后重试
3. 连续失败 3 次 → 在 heartbeat-panel.md 标记 "⚠️ 异常"
4. 继续尝试（不放弃轮询）
```

### 文件读取失败

```
1. 记录错误到日志
2. 跳过本次循环的处理步骤
3. 继续下一次轮询
4. 连续失败 3 次 → 在 heartbeat-panel.md 标记 "⚠️ 异常"
```

## 🔀 混合模式策略

不是所有 Agent 都需要同时心跳。根据职责特征分类：

| 模式 | Agent | 心跳类型 | 轮询间隔 | 启动时机 |
|------|-------|---------|---------|---------|
| 🔴 常驻心跳 | Coordinator | 始终轮询 | 240 秒 | ValveOS 激活后立即启动 |
| 🟡 按需心跳 | Worker | 有任务时轮询 | 400 秒 | Coordinator 分配任务时启动 |
| 🟢 唤醒模式 | Planner, PR Manager | 不轮询 | — | 用户手动唤醒 |
| 🔄 周期心跳 | Maintainer, Housekeeper | 每 N 轮执行一次巡检 | 800 秒 | 定时或触发条件满足时 |
| 🎯 监督心跳 | COO | 监控异常通知 | 640 秒 | ValveOS 激活后可选启动 |

### 模式选择指南

**何时启动心跳模式？**
- 需要 Agent 间实时通信 → 启动常驻心跳
- 有明确的任务需要执行 → 启动按需心跳
- 需要定期巡检 → 启动周期心跳
- 只需要一次性决策 → 使用唤醒模式（不启动心跳）

**何时停止心跳模式？**
- 任务完成且无后续任务 → 停止按需心跳
- 用户主动停止 → 发送 shutdown 消息
- 上下文接近耗尽 → 重启新会话（上下文会自动压缩，通常无需担心）

## 📋 心跳指令模板格式

心跳指令模板是用户粘贴到新会话的完整指令，格式如下：

```markdown
# [Agent名] 心跳模式

你是 [Agent全称]，现在进入心跳模式。

## 身份确认
- 角色：[Agent全称]
- Inbox：tasks/shared/inbox/[agent].md
- 心跳类型：[常驻/按需/周期/监督]
- 轮询间隔：[N] 秒

## 执行步骤
1. 执行命令：Start-Sleep -Seconds [N]（只执行这一条命令）
2. 读取你的 inbox：tasks/shared/inbox/[agent].md
3. 检查是否有未处理消息（没有 ✅ 标记的 📨 消息）
4. 如果有 → 处理消息 → 在消息头部添加 ✅ → 如需回复则追加到相关 inbox
5. 更新心跳面板：tasks/shared/heartbeat-panel.md（你的心跳计数+1，状态更新）
6. 回到步骤 1（不要再向我确认！无限循环！）

## 约束
- 绝对禁止使用 while/for 循环
- 绝对禁止挂到后台运行
- 只使用给定的 Sleep 命令
- 如果用户直接在会话中输入指令，立即响应
- 如果收到 shutdown 类型消息，停止轮询并报告

## 你的职责
[该 Agent 在心跳模式下的具体职责描述]
```

## 📊 心跳控制面板

> 详见 `tasks/shared/heartbeat-panel.md`

心跳面板是所有心跳 Agent 的状态看板，由 Agent 自行更新。

### 面板格式

| Agent | 状态 | 心跳计数 | 最后活跃时间 | 备注 |
|-------|------|---------|-------------|------|
| Coordinator | 🌙 Dormant | 0 | — | — |
| Worker | 🌙 Dormant | 0 | — | — |
| Planner | 🌙 Dormant | 0 | — | — |
| PR Manager | 🌙 Dormant | 0 | — | — |
| Maintainer | 🌙 Dormant | 0 | — | — |
| Housekeeper | 🌙 Dormant | 0 | — | — |
| COO | 🌙 Dormant | 0 | — | — |

### 更新规则

1. 每次轮询循环完成后，Agent 更新自己的行
2. 状态列使用 emoji：🌙 Dormant / 💓 Heartbeat / ⚡ Working / 💤 Standby
3. 心跳计数递增
4. 最后活跃时间使用 `$NOW` 变量
5. 备注列用于异常标记（⚠️ 异常）
