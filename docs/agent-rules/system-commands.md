# 系统命令（斜杠命令协议）

> 系统级协议，不属于任何单一 Agent。任何 AI 会话收到 `/` 开头的输入都必须遵循本协议。
>
> **唯一入口**：`/` 前缀 = 命令模式。AGENTS.md 负责信号检测，本文件负责注册、匹配与执行。

---

## 📡 斜杠命令协议

### 触发条件（铁律）

用户输入以 `/` 开头时，进入命令模式。无例外。

⚠️ **全程零元叙述**：从检测到 `/` 到输出最终结果的整个过程中，不输出任何非结果文字。包括但不限于：准备文字（"让我先…"）、过程描述（"正在读取…"、"现在写入…"）、完成声明（"命令日志已记录"、"已完成"）、以及结果后的追加文字（解释、指引、操作建议）。用户看到的应只有命令结果本身。

### 命令注册表

| 命令 | 别名 | 类别 | 匹配关键词 | 说明 |
|------|------|------|-----------|------|
| `/status` | `/s`, `/状态` | 信息 | 状态, 进度, status, 当前 | 查看系统状态 |
| `/reset` | `/r`, `/重置` | 操作 | 重置, reset, 从头开始, 清理 | 系统重置（需确认） |
| `/audit` | `/a`, `/审计` | 审计 | 审计, audit, 一致性, 检查 | 审计系统一致性 |
| `/help` | `/h`, `/?` | 信息 | 帮助, help, 命令, 列表 | 列出所有可用命令 |
| `/log` | `/l`, `/日志` | 信息 | 日志, log, 历史, 流水账 | 查看命令历史 |
| `/rehearsal-review` | `/rr`, `/演练复盘` | 审计 | 演练, 复盘, review, 测试结果 | 汇总演练报告 |
| `/workflow` | `/wf`, `/工作流` | 操作 | 工作流, 流程, workflow | 执行预定义工作流 |

### 协议流程

1. **检测** — 输入以 `/` 开头 → 进入命令模式
2. **提取** — 取 `/` 后的第一个词作为候选命令名
3. **精确匹配** — 在注册表中查找命令名或别名 → 命中则执行
4. **模糊匹配** — 未命中时，对 `/` 后的完整文本匹配关键词字段 → 高置信度直接执行，低置信度输出候选列表让用户确认
5. **未知命令** — 完全无法匹配 → 输出 `/help` 内容 + 提示
6. **记录（必须执行）** — 追加到 `tasks/logs/system-commands.log`，格式：

**记录范围**：
- `[INPUT]` — 用户触发的斜杠命令（如 `/status`、`/workflow`）
- `[WAKEUP]` — 用户触发的 Agent 唤醒动作（如 "唤醒 Planner"）
- `[RESPONSE]` — 命令处理完成后的响应摘要

**格式**：
```
[YYYY-MM-DD HH:MM:SS] [INPUT] "/用户输入" → 触发 /命令名
[YYYY-MM-DD HH:MM:SS] [RESPONSE] 处理完成，响应摘要
[YYYY-MM-DD HH:MM:SS] [WAKEUP] "唤醒 [Agent名]" → 触发 [Agent] 醒来协议
```
⚠️ **铁律**：每次执行斜杠命令或唤醒 Agent 后必须写入此日志文件，不可跳过。
⚠️ **时间纪律**：时间戳必须用 `Get-Date -Format "yyyy-MM-dd HH:mm:ss"` 获取真实系统时间，禁止编造。

### 兼容层（过渡期）

如果用户没有使用 `/` 前缀，但输入包含明确的系统命令关键词（如"执行系统重置""帮我审计"），仍可触发对应命令。
→ 触发时提示：``💡 建议使用斜杠格式，如 `/reset` 或 `/audit`，更精准且可扩展。``
→ 此兼容层将在未来版本移除。

## 📋 命令详情

### 1. /status — 查看状态

#### 执行步骤

1. 读取 `tasks/shared/agent-status.md` — Agent 状态 + 当前迭代任务看板
2. 读取 `tasks/shared/iteration-log.md` — 当前迭代字段（状态/任务数/已完成）
3. 运行 `git status` — Git 状态

#### 输出格式（必须）

```markdown
## 当前状态

### 迭代状态: Iteration [N]
- **状态**: [进行中/已完成/已废弃/待开始]
- **任务**: [X] 个任务，[Y] 已完成
- **当前任务**: [进行中任务的描述，如无则写"无"]

### Agent 状态
[从 agent-status.md 表格中读取]

### Git 状态
- **分支**: [当前分支]
- **未暂存改动**: [数量] 个（[列出文件名]）
- **未追踪文件**: [如有]

### 下一步建议
[动态推荐，按优先级从高到低匹配，只输出第一条命中的建议]
1. 有 `in_progress` 任务且对应 Worker 心跳超时（30分钟无更新）→ "⚠️ Worker-XXX 心跳超时，建议唤醒 Coordinator 检查"
2. 有 pending 任务且无人认领 → "📋 有 N 个待分配任务，建议唤醒 Coordinator"
3. 有 Agent 状态为"待机"且其 inbox 有未读消息 → "📨 [Agent名] 有未读消息，建议唤醒"
4. 所有任务 completed → "✅ 当前迭代任务已全部完成，建议唤醒 Planner 制定新计划"
5. 无迭代 → "请唤醒 Planner 启动新迭代"
```

#### ⚠️ 准确性铁律

- 迭代状态**必须与实际一致**
- Agent 状态从 `agent-status.md` 如实读取，不推测、不编造
- **结果边界**：输出上述状态面板后立即停止，不追加任何额外文字

### 2. /reset — 系统重置

#### 执行协议

详见 `cli-operations.md#系统重置`

#### 确认步骤

实际修改文件前，输出确认清单并等待用户回复：

```
⚠️ 即将执行完全重置，将执行以下操作：
  1. 清空所有 inbox（7个文件）
  2. 重置 agent-status.md 所有Agent为未启动
  3. 归档 Iteration X → 新建 Iteration X+1
  4. 清空 queue.md/assignments.md/observations.md 等

确认执行？回复"确认"开始。
```

#### 输出格式（执行完成后）

```markdown
## 重置完成

- **操作类型**: 完全重置
- **归档迭代**: Iteration [X] → 已废弃
- **新迭代**: Iteration [X+1] 就绪
- **已清空**: inbox（7个）/ queue.md / observations.md / agent-status.md

请唤醒 Planner 开始新迭代。
```

> ⚠️ **结果边界**：输出重置结果后立即停止。不追加"现在可以…"等文字。

### 6. /rehearsal-review — 演练复盘

> **数据源优先级（系统日志主导）**：
> 1. **P0 必须**：`tasks/shared/session-reports/*.md` — 所有 Agent 会话报告
> 2. **P0 必须**：`tasks/logs/system-commands.log` — 命令执行记录
> 3. **P1 推荐**：`tasks/logs/*.log` — 各 Agent 运行日志
> 4. **P2 可选**：遥测数据（用户主动提供时使用，不提供则跳过）
>
> 无遥测数据时仍可基于系统日志完成基本复盘。遥测数据仅用于补充细节（如元叙述违规的具体文本）。

#### ⚠️ session-report 有效性校验

读取 session-reports 时，**先验**报告格式再决定如何使用：

**旧格式（无效）**：单行表格格式 `| 时间 | 触发 | 动作摘要 | 阻塞 | 建议 |` — **无** 客观事实字段
- → 标记为 **⚠️ 日志不可信**，降级为"仅能确认 Agent 是否执行过"
- → **无法**验证：开场白合规、铁门协议合规、零元叙述合规
- → 继续读取 `.log` 文件补充

**新格式（有效）**：含协议合规节 + 4 个客观事实字段
- → 正常进行逐字段验证
- → 可直接判断：actual_first_output / pre_opening_exists / opening_verbatim_match / iron_door_compliance

**判定标准**：无 `actual_first_output` 字段 = 旧格式 = 日志不可信。

#### 执行步骤

1. 读取 `tasks/shared/session-reports/` 下所有 7 个文件 → 提取报告
2. 读取 `tasks/logs/system-commands.log` → 提取命令执行记录
3. （可选）如果用户提供了遥测数据路径 → 读取遥测文件补充细节
4. 从 session-reports 提取所有"发现的问题"条目和"协议合规"未通过项
5. 汇总为问题清单（按 P0/P1/P2 排序）
6. 对每个问题给出修复建议
7. 输出汇总报告

#### 输出格式

```markdown
## 演练复盘报告

### 问题清单

| 严重程度 | 问题 | 来源Agent | 修复建议 |
|----------|------|-----------|----------|
| P0 | [描述] | [Agent名] | [建议] |
| P1 | [描述] | [Agent名] | [建议] |

### 协议合规汇总

| 检查项 | 通过 | 未通过 |
|--------|------|--------|
| 角色切换 | N | N |
| 铁门协议 | N | N |
| 日志记录 | N | N |
| 状态更新 | N | N |

### 改进建议汇总
- [建议列表]

### 下一步
[如需修复 → "请唤醒 [Agent名] 执行修复"；如全部通过 → "✅ 演练通过，系统可进入正式运行"]
```

> ⚠️ **结果边界**：输出复盘报告后立即停止。

### 7. /workflow — 预定义工作流

#### 执行步骤

> ⚠️ **直接输出结果**：检测到 `/workflow` 后直接输出工作流列表或唤醒序列，禁止输出"让我先…"、"我来读取…"等准备文字。

1. 解析参数：提取工作流名称
2. 无参数 → 列出可用工作流
3. 有参数 → 匹配工作流名，输出唤醒序列

#### 可用工作流

| 工作流名 | 唤醒序列 | 说明 |
|----------|----------|------|
| `full-pipeline` | Planner → Coordinator → Worker → PR Manager | 完整流水线（从规划到PR） |
| `sync-upstream` | Planner → Worker | 同步上游（分析+执行） |
| `cleanup` | Housekeeper → Maintainer | 仓库清理+分析 |
| `audit` | COO | 系统审计 |

#### 输出格式（无参数）

```markdown
## 可用工作流

| 工作流 | 唤醒序列 | 说明 |
|--------|----------|------|
| full-pipeline | Planner → Coordinator → Worker → PR Manager | 完整流水线 |
| sync-upstream | Planner → Worker | 同步上游 |
| cleanup | Housekeeper → Maintainer | 仓库清理 |
| audit | COO | 系统审计 |

使用方式：`/workflow full-pipeline`
```

#### 输出格式（有参数）

```markdown
## 工作流: [工作流名]

**唤醒序列**：
1. 唤醒 [Agent1] — [原因]
2. 唤醒 [Agent2] — [原因]
...

请按顺序执行，每步完成后唤醒下一个 Agent。
```

> ⚠️ **结果边界**：输出上述内容后立即停止。不追加"根据工作流…"、"准备好后…"、"请唤醒 X 开始"等任何额外文字。
