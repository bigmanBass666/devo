## 1. 背景与动机

### 1.1 问题陈述

**当前问题**：ValveOS 作为全局默认的多 Agent 协作系统，在每次会话中都被强制加载，导致：

- **过度服务**：70%+ 的会话不需要多 Agent 协作（如快速提问、GitHub Actions 调试、代码解释）
- **性能开销**：每次会话都要读取完整的 AGENTS.md（139行）、执行角色切换协议、检查 inbox 等
- **用户体验差**：简单任务被迫进入复杂流程，增加认知负担
- **设计哲学违背**：ValveOS 定位为"国之重器"，但使用频率过高导致贬值

### 1.2 用户洞察

**核心理念**（来自用户）：
> "ValveOS 不是默认项，它更像是一个可选的开关，是一件国之重器，默认是不打开的才对。"

**关键场景**：
- 用户打开新会话解决 GitHub Actions 问题 → 不需要 ValveOS
- 用户想快速重构一个函数 → 不需要 ValveOS
- 用户要提一个完整 PR → **需要** ValveOS
- 用户要做系统审计 → **需要** ValveOS

### 1.3 设计目标

| 目标 | 描述 | 优先级 |
|------|------|--------|
| **按需激活** | ValveOS 必须通过 `/valveos` 显式命令才能启用 | 🔴 必须 |
| **轻量默认** | 普通模式零协议开销，直接对话 | 🔴 必须 |
| **安全底线** | 关键安全规则始终生效 | 🔴 必须 |
| **智能提示** | AI 检测到复杂任务时主动询问是否启动 | 🟡 推荐 |
| **平滑迁移** | 向后兼容现有 AGENTS.md 内容 | 🟡 推荐 |

- 永不删除 main 分支
- 永不 push 到 upstream（只读）

## 3. 模式切换机制

### 3.1 触发方式

#### 方式 1：显式命令（唯一可靠的方式）

| 命令 | 效果 | 示例 |
|------|------|------|
| `/valveos` 或 `/vo` | 进入 ValveOS 模式 | "我想用 ValveOS 处理这个 PR" |
| `唤醒 [Agent名]` | 进入 ValveOS + 直接唤醒指定 Agent | "唤醒 Coordinator 分配任务" |

**⚠️ 重要**：只有这两种方式能激活 ValveOS。其他任何表述都不会触发。

#### 方式 2：AI 主动推荐（辅助）

当 AI 在普通模式下检测到以下条件时，**主动询问**用户：

```markdown
## 触发条件（满足任一即触发）

### 条件 A：任务复杂度
- 任务步骤数 ≥ 3
- 涉及 ≥ 2 个专业领域（编码 + 测试 + 文档）
- 预估耗时 > 30 分钟

### 条件 B：关键词匹配
用户输入包含："PR 流程"、"多 Agent"、"协作"、"分工"、"并行"

### 条件 C：上下文暗示
- 用户提到"需要追踪状态"
- 用户提到"需要多个角色配合"
- 任务明显属于 ValveOS 典型场景（见下方）
```

**AI 推荐话术模板**：

```markdown
🤔 **检测到复杂任务**

这个任务涉及 [N] 个步骤和 [M] 个专业领域。

**选项 A**: 我直接帮你完成（🌿 普通模式 - 快速但单打独斗）
**选项 B**: 启动 ValveOS 多 Agent 协作（⚡ 结构化、可追踪、专业化）

你选哪个？(A/B)
```

**⚠️ 关键约束**：
- AI **只能询问**，不能自动激活
- 用户必须明确选择 B 或输入 `/valveos`
- 如果用户选择 A 或忽略，继续普通模式

### 3.2 模式行为对比

| 维度 | 🌿 普通模式 | ⚡ ValveOS 模式 |
|------|-------------|----------------|
| **启动方式** | 默认 | `/valveos` 或 `唤醒 X` |
| **文件加载** | AGENTS.md (~40行) | AGENTS.md + VALEVEOS.md (~250行) |
| **AI 身份** | 通用助手 | 可切换为指定 Agent |
| **任务执行** | 单 Agent 直接完成 | 多 Agent 协调分配 |
| **通信机制** | 无 | Inbox 文件通信 |
| **状态追踪** | 无 | agent-status / iteration-log |
| **日志记录** | 无 | 完整日志系统 |
| **审计能力** | 无 | valveos-audit skill |
| **适用场景** | 快速提问、单一任务、调试排查 | PR 流程、系统维护、多步协作 |
| **开销** | 极低 | 较高（但值得） |

## 4. 规则分层详细说明

### 4.1 为什么这样分层？

**原则**：不是所有规则都同等重要。

| 规则类型 | 重要性 | 后果 if 违反 | 应该何时提醒 |
|----------|--------|--------------|--------------|
| **安全铁律** | 🔴 致命 | 可能破坏仓库、泄露密钥 | **每时每刻** |
| **提交纪律** | 🟠 重要 | Git 历史混乱、协作困难 | **每次操作 Git 时** |
| **社交边界** | 🟡 中等 | 可能越权代表用户 | **涉及外部操作时** |
| **ValveOS 协议** | 🔵 特定 | 仅影响 ValveOS 内部流程 | **仅 ValveOS 模式** |

### 4.2 全局规则的"最小必要集"

经过审计，以下规则**必须在所有模式下生效**：

#### ✅ 规则 1：安全铁律（5 条）

```markdown
## 🔒 安全铁律

1. 永不删除 main 分支
2. 永不 push 到 upstream（只读远程）
3. 永不向 PR 中包含 tasks/、notifications/、.trae/、AGENTS.md
4. 永不绕过用户直接操作 origin 以外的远程
5. 重置前必须二次确认
```

**理由**：这些是仓库安全的底线，无论做什么任务都不能违反。

#### ✅ 规则 2：提交纪律（1 条）

```markdown
## 📝 提交纪律

每次更改后立即 `git add` + `git commit` + `git push`，格式 `type: 描述`。
```

**理由**：Git 操作规范应该统一，避免脏工作区。

#### ✅ 规则 3：社交边界（简化版）

```markdown
## 🛡️ 社交边界（基础版）

- **可自主**：本地代码修改、测试、分析、提交、读取通知、运行构建
- **不可自主**：回复评论、创建/更新 PR/issue、任何代表用户的行为、合并到上游
- **技术决策**：Agent 分析推荐，用户批准；主动提选项而非等待指令
```

**理由**：防止 AI 越权，但在普通模式下可以更灵活（例如用户可能就是想让 AI 回复评论）。

**⚠️ 注意**：在 ValveOS 模式下，社交边界会更严格（见 VALEVEOS.md）。

### 4.3 ValveOS 专属规则（移至 VALEVEOS.md）

以下规则**仅在 ValveOS 模式下才有意义**：

| 规则 | 原因 |
|------|------|
| 铁门协议 | 普通 Agent 间无通信需求 |
| 角色切换协议 | 普通模式不切换角色 |
| 系统命令（/status, /audit 等） | 这些命令操作 ValveOS 状态 |
| 路由表 | 路由目标是 ValveOS 内部组件 |
| 应急手册 | 处理 ValveOS 特定故障 |
| 时间纪律 | ValveOS 日志系统专用 |
| 审计机制 | 仅 ValveOS 有审计需求 |

## 6. 迁移策略

### 6.1 渐进式迁移（推荐）

**阶段 1：准备（本次实施）**
1. 创建 `docs/agent-rules/valveos-protocol.md`（从 AGENTS.md 提取内容）
2. 精简 AGENTS.md 为基础规则版本
3. 在 AGENTS.md 顶部添加模式说明
4. 测试两种模式的加载和行为

**阶段 2：验证（1-2 天）**
1. 在日常工作中同时使用两种模式
2. 收集反馈：
   - 普通模式是否足够轻量？
   - ValveOS 模式是否功能完整？
   - 切换是否流畅？
3. 修复发现的问题

**阶段 3：固化（验证通过后）**
1. 将 `valveos-protocol.md` 纳入正式文档体系
2. 更新 SYSTEM-MANIFEST.md 的 File Registry
3. 执行 valveos-audit 确保一致性
4. 归档旧版 AGENTS.md（如果需要）

### 6.2 向后兼容性保证

**关键承诺**：
- ✅ 现有 AGENTS.md 的所有内容都保留（只是移动位置）
- ✅ 所有 `/` 命令继续工作（只是在 ValveOS 模式下）
- ✅ `唤醒 X` 继续工作（并自动激活 ValveOS）
- ✅ 现有的 inbox、agent-status 等文件不受影响
- ✅ 已有的 session-reports 和日志保持有效

**唯一变化**：
- ❌ 不再默认加载 ValveOS 协议（这是目的）

## 8. 风险与缓解

### 8.1 潜在风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 用户忘记输入 `/valveos` 导致无法启动 ValveOS | 中 | 高 | AI 主动推荐机制（方式 2） |
| 精简后的 AGENTS.md 遗漏重要规则 | 低 | 高 | 严格审计 + 测试覆盖 |
| 迁移过程中出现不一致 | 中 | 中 | 渐进式迁移 + 每步验证 |
| 现有工作流被破坏 | 低 | 高 | 向后兼容性保证 |
| 用户对新模式感到困惑 | 中 | 中 | 清晰的文档 + 过渡期支持 |

### 8.2 成功标准

如何判断这次设计成功？

**定量指标**：
- [ ] AGENTS.md 从 139 行减少到 <50 行（减少 65%+）
- [ ] 普通模式下的首次响应时间缩短（目标：<2 秒 vs 当前可能 >5 秒）
- [ ] ValveOS 模式的功能完整性 100% 保持（所有测试通过）

**定性指标**：
- [ ] 用户反馈："现在简单任务快多了"
- [ ] 用户反馈："ValveOS 启动很清晰，不会误触"
- [ ] 团队成员理解双层架构的设计理念

### 9.2 设计原则总结

本次设计遵循的核心原则：

| 原则 | 应用 |
|------|------|
| **YAGNI** | 不过度设计，只解决当前问题 |
| **最小惊讶** | 默认行为符合直觉（普通 = 简单） |
| **渐进增强** | 从简单开始，按需加载复杂功能 |
| **安全第一** | 关键规则始终生效，不被绕过 |
| **用户控制** | 用户决定何时使用重武器 |

## 附录 A：FAQ

**Q1: 这会不会让 ValveOS 很难被发现？**
A: 不会。AGENTS.md 顶部会有醒目的入口指引，而且 AI 会在合适时机主动推荐。

**Q2: 现有的 `/status`、`/audit` 等命令还能用吗？**
A: 能，但只在 ValveOS 模式下。普通模式下这些命令没有意义（因为没有状态可查）。

**Q3: 如果我在普通模式下想临时用一个 Agent 的能力怎么办？**
A: 输入 `唤醒 [Agent名]` 即可，这会自动激活 ValveOS 并切换到该 Agent。

**Q4: 这个改动会影响已有的 session-reports 和日志吗？**
A: 不会。所有历史数据保持不变，只是未来的会话会区分模式。

**Q5: 能否回退到旧的全局默认模式？**
A: 能。只需将 AGENTS.md 恢复为完整版，删除 valveos-protocol.md 即可。但我们不建议这样做。

| 术语 | 定义 |
|------|------|
| **普通模式** (Normal Mode) | 默认的轻量 AI 对话模式，只加载基础规则 |

| **按需激活** (On-Demand Activation) | 只有用户明确请求才加载重型系统的策略 |
| **显式命令** (Explicit Command) | `/valveos` 或 `唤醒 X` 这类无歧义的触发方式 |
| **主动推荐** (Proactive Suggestion) | AI 检测到复杂任务时询问是否启动 ValveOS |
| **国之重器** (Heavy Weapon) | 用户对 ValveOS 的定位——强大但不常用 |
