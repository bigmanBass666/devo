# 技术决策记录

> 来源：Anchor 系统反哺（trae-unlock 项目）

### [2026-04-20 13:00] 为什么去品牌化

**选择**: 品牌名只在 AGENTS.md + SYSTEM-MANIFEST.md 中硬编码，其他文件去品牌化
**否决**: 全项目统一使用品牌名（~18 个文件 ~48 处）
**原因**: Anchor 项目的教训——品牌名散布在所有文件中，改名时"牵一发动全身"。集中定义后，改名只需修改 AGENTS.md（2 处）+ SYSTEM-MANIFEST.md（2 处），共 4 处。shared/*.md 和 instructions.md 只描述功能不提品牌名。

**改名规则**:
- **活文件** (AGENTS.md, SYSTEM-MANIFEST.md): 集中定义品牌名，改名必改
- **shared/*.md**: 去品牌化，只描述功能，改名不需要动
- **instructions.md**: 去品牌化，用"本系统"代替品牌名
- **历史文件** (audit-log.md, iteration-log.md): 保持原名，改名等于篡改历史

### [2026-04-19 23:20] 为什么 AGENTS.md 从静态定义改为路由表

**选择**: AGENTS.md 路由化（触发→动作→速查）
**否决**: 保持静态信息声明式写法
**原因**: AGENTS.md 通过 trae_rules_context 注入到每次 AI 回复，是唯一保证始终在上下文内的文件。利用这个"每次必读"特性，将信息声明型章节改为触发→动作型路由，让 AI 在遇到特定场景时知道去哪找答案。实测验证：路由对"操作型指令"有效（系统重置成功触发路由），对"查询型指令"效果较弱。

### [2026-04-20 13:00] 为什么闭环是分形的

**选择**: 不硬编码闭环的范围边界，保持闭环机制的可扩展性
**否决**: 假设闭环只在当前项目内
**原因**: Anchor 项目的教训——他们解决了项目级闭环（同一项目跨会话），但无意中硬编码了闭环的边界，后来不得不推翻重设计。闭环是分形的：会话级 → 项目级 → 跨项目级 → 更高级别。今天解决项目级，明天可能需要跨项目级。不要硬编码闭环的范围。

### [2026-04-19 21:37] 为什么引入 System Manifest 元数据层

**选择**: 新建 SYSTEM-MANIFEST.md 作为元数据唯一事实来源
**否决**: 继续在 ~10 处文件中重复声明 Agent 列表/架构模型
**原因**: "全硬编码"问题——每次新增 Agent 需要改 12+ 个文件。引入 Manifest 后，新增 Agent 从"改 12 个文件"降为"改 Manifest 1 处 + 审计辅助"。

### [2026-04-20 04:46] 为什么系统重置时保护审计日志

**选择**: audit-log.md 归档保留（添加重置分隔线，不清空历史）
**否决**: 清空 audit-log.md（只保留表头）
**原因**: 实测发现系统重置会清空整个迭代的审计记录。审计日志是跨迭代的制度记忆，不是运行时状态。运行时数据（inbox/queue/status）可以清空，但制度记忆必须保留。
