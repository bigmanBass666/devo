### [2026-04-20 13:00] 为什么去品牌化

**选择**: 品牌名只在 AGENTS.md + SYSTEM-MANIFEST.md 中硬编码，其他文件去品牌化
**否决**: 全项目统一使用品牌名（~18 个文件 ~48 处）
**原因**: Anchor 项目的教训——品牌名散布在所有文件中，改名时"牵一发动全身"。集中定义后，改名只需修改 AGENTS.md（2 处）+ SYSTEM-MANIFEST.md（2 处），共 4 处。shared/*.md 和 instructions.md 只描述功能不提品牌名。

**改名规则**:
- **活文件** (AGENTS.md, SYSTEM-MANIFEST.md): 集中定义品牌名，改名必改
- **shared/*.md**: 去品牌化，只描述功能，改名不需要动
- **instructions.md**: 去品牌化，用"本系统"代替品牌名
- **历史文件** (audit-log.md, iteration-log.md): 保持原名，改名等于篡改历史

### [2026-04-19 23:20] 为什么 AGENTS.md 从静态定义改为路由表

**选择**: AGENTS.md 路由化（触发→动作→速查）
**否决**: 保持静态信息声明式写法
**原因**: AGENTS.md 通过 trae_rules_context 注入到每次 AI 回复，是唯一保证始终在上下文内的文件。利用这个"每次必读"特性，将信息声明型章节改为触发→动作型路由，让 AI 在遇到特定场景时知道去哪找答案。实测验证：路由对"操作型指令"有效（系统重置成功触发路由），对"查询型指令"效果较弱。

### [2026-04-20 13:00] 为什么闭环是分形的

**选择**: 不硬编码闭环的范围边界，保持闭环机制的可扩展性
**否决**: 假设闭环只在当前项目内
**原因**: Anchor 项目的教训——他们解决了项目级闭环（同一项目跨会话），但无意中硬编码了闭环的边界，后来不得不推翻重设计。闭环是分形的：会话级 → 项目级 → 跨项目级 → 更高级别。今天解决项目级，明天可能需要跨项目级。不要硬编码闭环的范围。

---

### [2026-04-19 21:37] 为什么引入 System Manifest 元数据层

**选择**: 新建 SYSTEM-MANIFEST.md 作为元数据唯一事实来源
**否决**: 继续在 ~10 处文件中重复声明 Agent 列表/架构模型
**原因**: "全硬编码"问题——每次新增 Agent 需要改 12+ 个文件。引入 Manifest 后，新增 Agent 从"改 12 个文件"降为"改 Manifest 1 处 + 审计辅助"。

### [2026-04-20 04:46] 为什么系统重置时保护审计日志

**选择**: audit-log.md 归档保留（添加重置分隔线，不清空历史）
**否决**: 清空 audit-log.md（只保留表头）
**原因**: 实测发现系统重置会清空整个迭代的审计记录。审计日志是跨迭代的制度记忆，不是运行时状态。运行时数据（inbox/queue/status）可以清空，但制度记忆必须保留。

### [2026-04-20 21:00] 为什么选择斜杠命令协议（/ 前缀）

**选择**: 用户输入以 `/` 开头进入命令模式，支持 `/+自然语言` 模糊匹配
**否决**: 在 AGENTS.md 中列出每条命令的自然语言触发词列表
**原因**: 
1. 触发词分散在 AGENTS.md 和 system-commands.md 两处，违反单一事实来源原则
2. 命令增多时 AGENTS.md 必然线性膨胀（从 3 条到 N 条）
3. `/` 前缀是成熟模式（Claude Code / GitHub CLI / Cursor 都用类似设计）
4. "永恒上下文"洞察：AGENTS.md 应只做"判断场景→指路"，不列出具体功能细节
**关键决策点**: 兼容层保留过渡期友好性（无 `/` 但含明确关键词仍可触发），未来移除

### [2026-04-20 21:00] 为什么需要 AGENTS.md 安全铁律

**选择**: 在 AGENTS.md 中添加 5 条不可违反的安全规则（不删 main、不 push upstream 等）
**否决**: 将安全规则分散在各 instructions.md 中
**原因**:
1. 安全规则是跨所有 Agent 的全局不变量，不是某个 Agent 的职责
2. 利用 AGENTS.md"每次被读取"特性 = 每次操作前都看到安全提醒
3. 与"行为常量层"定位一致——AGENTS.md 承载的是永远不该变的东西
**类比**: 类似操作系统的内核态权限检查——在进入用户空间之前先验证

### [2026-04-20 21:00] 为什么需要项目理解文档（project-understanding.md）

**选择**: 创建 tasks/shared/project-understanding.md 记录上游代码结构映射
**否决**: 让 Worker 每次自己探索代码库
**原因**:
1. ValveOS 的 Worker 是纯文档驱动的 AI，没有 IDE 的代码索引能力
2. 盲目搜索整个 workspace 效率低且可能遗漏关键模块
3. 项目理解文档是"地图"——Worker 可以快速定位"改配置在哪""加工具在哪"
4. 这是 ValveOS 与上游代码之间的唯一桥梁，填补了架构视角与代码现实之间的鸿沟

### [2026-04-20 21:00] 为什么 4 个横切服务 Agent 缺少边界条件章节

**发现**: PR Manager、Maintainer、Housekeeper、COO 的 instructions.md 缺少显式的"边界条件"章节
**根因分析**:
1. 这 4 个 Agent 是"横切服务"——不直接参与主工作流（规划→协调→编码→PR）
2. 编写 instructions 时主要关注正常流程，对异常路径覆盖不足
3. Planner/Coordinator/Worker 因为主流程高频使用，经过更多迭代自然补全了边界处理
**修复方案**: 为 4 个 Agent 各补充"无消息时/失败时/异常时"三个维度的边界条件，格式统一

### [2026-04-21 01:00] 为什么需要角色切换协议（Rehearsal 1 发现）

**选择**: 用户说"唤醒 [Agent名]"时，AI 立即切换到该 Agent 角色执行
**否决**: 写入 inbox 等待下次会话
**原因**:
1. Rehearsal 1 实测：用户说"唤醒 Planner"后，AI 仍然以根 Agent 身份对话，没有变成 Planner
2. 根因：AGENTS.md 和 instructions.md 没有定义"当用户说唤醒 X 时，AI 应立即切换角色"的协议
3. 没有角色切换，多 Agent 系统无法运转——用户永远在和调度器说话，Agent 永远不会被真正激活
4. 铁门协议补充：用户直接唤醒时不需要 inbox 中转（inbox 用于 Agent 间异步通知）
**类比**: 操作系统的 exec()——进程直接替换，不是 fork()+wait()

### [2026-04-21 01:00] 为什么废除轮询待机模式（Rehearsal 1 发现）

**选择**: 待机 = 状态标记 + 断点续传（无后台进程）
**否决**: Start-Sleep 轮询 inbox
**原因**:
1. Rehearsal 1 实测：Agent 用 Start-Sleep 尝试轮询，但 sleep 结束后 AI 不会自动醒来
2. 根因：文档驱动的 AI 系统没有后台进程能力。AI 会话是一次性的，不是持久进程
3. Start-Sleep 在 AI 上下文中只是"暂停然后结束"，不是"暂停然后继续"
4. while 循环更糟——被超时杀掉后 Agent 会话异常，上下文浪费
**影响范围**: cli-operations.md + 3 个 instructions.md（coordinator/workers/pr-manager）+ ARCHITECTURE.md + SYSTEM-MANIFEST.md + multi-agent-user-guide.md

### [2026-04-21 01:00] 为什么斜杠命令日志必须强制写入（Rehearsal 1 发现）

**选择**: 在协议流程第 6 步添加显式写入要求 + 格式模板 + 铁律标记
**否决**: 仅声明"执行完毕后追加到日志"
**原因**:
1. Rehearsal 1 实测：用户执行了 /help、/status、/log，但 system-commands.log 仍为空
2. 根因：协议只说了"记录"，AI 没有主动执行写入操作
3. AI 对"记录"的理解是"在回复中提到"，不是"写入文件"
4. 必须用显式的格式模板 + 铁律标记消除歧义

### [2026-04-21 04:30] 为什么角色切换必须禁止元叙述（Rehearsal 2 发现）

**选择**: AI 第一句输出必须是 Agent 开场白，禁止描述切换过程
**否决**: 允许 AI 先说"我正在切换到 X 角色"
**原因**:
1. Rehearsal 2 实测：Worker 被唤醒时先说"我正在切换到 Worker 角色。让我先读取相关指令文件"
2. 元叙述破坏角色沉浸感——用户看到的是"AI 正在变成 Worker"而不是"Worker 醒来了"
3. 类比：exec() 替换进程后，新进程不会先打印"我正在替换旧进程"
4. 所有读取操作应在内部完成，用户只看到角色直接"醒来"

### [2026-04-21 05:10] 为什么必须强制使用 standard-openings.md 原文作为开场白（Rehearsal 3 发现）

**选择**: 第一句输出必须是 standard-openings.md 原文，不可改写
**否决**: 允许 AI 自创开场白（如"Coordinator 已就绪""Worker-001 已苏醒"）
**原因**:
1. Rehearsal 3 实测：4 个 Agent 中 3 个自创了开场白，没有使用 standard-openings.md
2. 元叙述禁令只禁止了"我正在切换"，AI 找到了其他绕过方式："X 已就绪""X 已苏醒""我现在以 X 身份"
3. 自创开场白 = 元叙述的变体——用户看到的是"AI 在宣告角色"而不是"角色在说话"
4. 强制原文消除了所有歧义：不存在"怎么改写才算合规"的问题
5. 类比：exec() 后新进程不会先打印"我已替换旧进程"，而是直接执行新代码

### [2026-04-21 05:30] 为什么开场白前必须绝对零输出（Rehearsal 4 发现）

**选择**: 标准开场白必须是 AI 的绝对第一句输出，之前零输出
**否决**: 允许 AI 在开场白前输出"准备工作"描述
**原因**:
1. Rehearsal 4 实测：Coordinator 在开场白前输出了"我需要先读取 Coordinator 的指令文件..."，Worker 输出了"I'll wake up the Worker agent..."
2. R3 的禁令说"第一句输出必须是标准开场白"，AI 理解为"第一句正式输出"，把元叙述当成了"准备工作"
3. 元叙述从"替代开场白"进化为"开场白前缀"——AI 在开场白之前插入了描述
4. 唯一解法：开场白前绝对零输出，无论中英文、无论什么形式
5. COO 在 R4 中完美执行（零元叙述），证明这是可行的

### [2026-04-21 06:00] 遥测数据中的空白行是工具调用，不是元叙述（Rehearsal 5 洞察）

**发现**: Housekeeper 遥测数据显示标准开场白前有约 18 行空白，初判为"开场白前输出空白行"违规
**修正**: 这些空白行实际上是 AI 的工具/命令调用（如 `git fetch origin`、`git branch -r`、读取文件等），在导出为 Markdown 时工具调用不显示内容，呈现为空白行
**关键洞察**: 遥测数据（聊天记录导出）中的空白行 ≠ AI 主动输出的空白行。AI 在开场白前执行读取操作是合法的（AGENTS.md 明确说"所有读取操作在内部静默完成"），这些操作在聊天记录中表现为空白
**影响**: P2-B（开场白前空白行）是误判，不需要修复。AGENTS.md 的"开场白前绝对零输出"规则本身是正确的，但判断违规时需要区分"工具调用产生的空白"和"AI 主动输出的空白"
**原则**: 演练复盘时，不能仅凭遥测数据的空白行判定违规，必须结合 AI 的实际输出文本判断

### [2026-04-21 06:30] 用阀门迭代阀门 — ValveOS 双重闭环（架构级洞察）

**发现**: 用户指出阀门（用户）不应做决策，只应授权。Agent 知识量远超用户，应自主判断全局最优解并执行，仅在需要阀门打开时通知。
**核心原则**:
1. **阀门不决策，阀门只执行指令**：AI 分析全局后给出一条明确指令（如"唤醒 Worker 做 XXX"），用户只需复制执行。不给选项，不给选择题
2. **AI 不替代 Agent 工作**：当前会话专注于系统迭代本身。需要其他 Agent 执行的任务 → 写入 inbox 或输出唤醒指令让用户触发新会话
3. **用阀门迭代阀门**：ValveOS 的运维过程本身也应闭环——系统自复盘、自迭代、自我增强
4. **双重闭环螺旋上升**：内环 = Agent 工作流闭环；外环 = ValveOS 系统自身迭代闭环；闭环范围从系统内扩展到整个项目
5. **降低用户信息理解密度**：AI 做完所有能做的决策，用户只需理解并执行一条指令
**影响**: 改变交互模式——"给选项让用户选"→"做决策给指令让用户执行"
**实施**: 从此刻起，当前会话的 AI 应自主确定最优路径，输出单条明确指令
